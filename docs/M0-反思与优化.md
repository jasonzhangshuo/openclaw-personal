# M0 反思与优化

## 1. 目标完成度与问题

### 1.1 M0 官方目标（蓝图）

- OpenClaw 可执行、版本记录
- Feishu 与 MemOS **插件安装成功**
- Gateway 可用
- 基线/配置可记录

### 1.2 实际暴露的问题

| 问题 | 原因 | 谁该做 |
|------|------|--------|
| 你填了很多东西、手动操作多 | 没有「只填一份 .env」的单一入口；步骤分散在 Runbook 多处 | 自动化：一份 .env.example + 一个脚本 |
| 1008 / token 找不到 | Dashboard 认证入口不直观；未用「带 token 的 URL」一步到位 | 自动化：脚本用 `?token=` 打开浏览器 |
| workspace 报错 mkdir '/workspace-main' | config 里用相对路径，Gateway 解析成系统根目录 | 自动化：脚本生成 config 时用项目内绝对路径 |
| .env 写了但 Chat 仍无回复 | Gateway 进程读的是 STATE_DIR 下的 .env，不是项目根 .env | 自动化：脚本统一把 .env 同步到 .openclaw/state/.env |
| 飞书等「还要配什么」不清晰 | M0/M1 边界和「你现在只需填啥」没说清 | 文档：M0=只填 LLM Key；M1=再填飞书三件套 |

### 1.3 边界澄清

- **M0**：环境 + 基线 + **本机 Chat 能回复**（只需 1 个 LLM Key）。
- **M1**：飞书单群打通（需要你提供：Feishu App ID、App Secret、测试群 ID）。

所以你**只需要提供**：  
- M0：一个 LLM 的 API Key（如 ZAI_API_KEY）。  
- M1：飞书应用的 appId、appSecret、一个群 ID。  

其余（安装、config 生成、workspace 路径、token、Gateway、Dashboard 打开）都应**由脚本 + 文档引导完成**，不让你到处找地方填、到处手动执行。

---

## 2. 优化原则

1. **单一入口**：只填一份 `.env`（复制 `.env.example` 后填空）。
2. **单一命令**：`./scripts/m0-bootstrap.sh` 做完所有能自动做的事。
3. **自然引导**：脚本开头检查「.env 里是否有至少一个 LLM Key」；没有则**只**提示「请填 .env 里的 xxx，然后重新运行本脚本」并退出。
4. **零相对路径**：config 里的 workspace、插件 path 等全部用项目绝对路径，避免 ENOENT。
5. **Dashboard 一步到位**：用 `open "http://127.0.0.1:18789/?token=…"`，不再要求你找设置页粘贴 token。

## 3. 当前仍缺的配置（检查结果）

- **飞书**：config 里仍是占位 `replace_app_id`、`replace_with_secret`、`oc_main_test_group`。属于 **M1**，不在 M0 必做范围内；若要在 M0 阶段就预留好，可在 .env.example 里列出 FEISHU_APP_ID、FEISHU_APP_SECRET、FEISHU_GROUP_ID_MAIN，脚本在「有这些变量时」自动写进 config。
- **MemOS**：已安装，enabled: false，M3 再开。无缺项。
- **LLM**：已能回复即说明已就绪。

## 4. 后续调整清单

- [x] 一份 `.env.example`：分块注明 M0 必填 / M1 飞书选填 / M3 MemOS 选填。

- [x] config 由脚本生成或校验：workspace 用绝对路径，gateway.mode、gateway.auth 存在，避免手写导致路径/缺项错误。

- [x] M1 飞书：若 .env 中已填 FEISHU_APP_ID / FEISHU_APP_SECRET / FEISHU_GROUP_ID_MAIN，脚本自动写入 config，用户再次运行本脚本即可。

## 5. 小结

M0 没做好的核心是：**自动化不彻底 + 引导不单一**。  
优化后：**你只提供 API Key（和可选飞书三件套），其余由一份 .env + 一条脚本完成，并明确区分 M0 / M1 需要你填的内容。**
